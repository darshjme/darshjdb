What is DarshanDB?

Every backend project starts the same way: three weeks of plumbing before you write a single line of business logic. Setting up Postgres. Writing REST APIs. Building auth. Wiring WebSockets. Handling file uploads. Managing permissions. DarshanDB eliminates all of that.

DarshanDB is a complete backend-as-a-service compiled into a single Rust binary you can run on a $5 VPS. It takes the best ideas from Firebase (real-time sync), Supabase (PostgreSQL foundation), InstantDB (declarative query language), and Convex (server functions) -- and combines them into one open-source, self-hosted system. No vendor lock-in, no cloud dependency, no compromises.

Built by Darsh Joshi from Ahmedabad, India. The name comes from the Sanskrit word darshan, meaning "vision" -- the ability to perceive the complete picture. DarshanDB sees every change, every query, every permission, and reactively pushes exactly the right data to exactly the right clients. Learn more at darshj.me.

Quick Start

Get a complete backend running in under five minutes. You need Docker (for Postgres) or an existing PostgreSQL 16+ instance, and Node.js 18+ for client SDKs.

  1. Install DarshanDB

    A single ~30MB binary. No runtime dependencies.

    bash 
    # macOS / Linux
curl -fsSL https://darshandb.dev/install | sh

# Or with Docker
docker compose up -d
  2. Start the Dev Server

    Auto-creates PostgreSQL, seeds the admin user, starts the dashboard.

    bash 
    darshan dev

# Dashboard  ->  http://localhost:7700/admin
# WebSocket  ->  ws://localhost:7700
# REST API   ->  http://localhost:7700/api/
  3. Install the SDK and Connect

    Pick your framework. Here is React:

    bash 
    npm install @darshan/react
    tsx 
    import { DarshanProvider, DarshanDB } from '@darshan/react';

const db = DarshanDB.init({ appId: 'my-app' });

function App() {
  return (
    <DarshanProvider db={db}>
      <TodoApp />
    </DarshanProvider>
  );
}

function TodoApp() {
  // This is a LIVE query -- updates when anyone changes data
  const { data, isLoading } = db.useQuery({
    todos: {
      $where: { done: false },
      $order: { createdAt: 'desc' },
      owner: {}  // fetch related user in one query
    }
  });

  if (isLoading) return <p>Loading...</p>;

  return (
    <ul>
      {data.todos.map(todo => (
        <li key={todo.id}>{todo.title}</li>
      ))}
    </ul>
  );
}

    That is it. Real-time sync, offline support, optimistic updates, type safety -- all from five lines of configuration.

Triple Store (EAV Architecture)

DarshanDB stores all data as triples: (entity, attribute, value). This is an Entity-Attribute-Value model built on top of PostgreSQL, giving you the flexibility of a document database with the power of relational queries.

How It Works

Instead of rigid tables with fixed columns, every piece of data is a triple. When you write { title: "Buy groceries", done: false } to a todos entity, DarshanDB stores three rows internally:

Entity IDAttributeValue
todo-abctitle"Buy groceries"
todo-abcdonefalse
todo-abccreatedAt1712345678000

Why Triples?

  • Schema-on-read in development -- Write first, schema emerges. No migrations needed while prototyping.
  • Strict mode for production -- Opt-in enforcement with auto-generated migrations when you are ready.
  • Graph traversal -- Relations between entities are just triples pointing to other entities, enabling nested queries.
  • Time-travel -- MVCC snapshots let you query any past state of your data.
  • Multi-tenancy -- Namespace isolation with shared infrastructure.
💡

Despite being stored as triples internally, you interact with DarshanDB using familiar JSON objects. The triple store is an implementation detail -- your application code looks like any modern database client.

DarshanQL

DarshanQL is a declarative, relational query language designed for client-side use. Every query is automatically a live subscription -- when data changes, your app updates instantly.

Basic Query

typescript 
// Fetch all todos
const { data } = db.useQuery({ todos: {} });

// Filter, sort, paginate
const { data } = db.useQuery({
  todos: {
    $where: { done: false, priority: { $gte: 3 } },
    $order: { priority: 'desc', createdAt: 'asc' },
    $limit: 10,
    owner: {}  // include the related user
  }
});

Nested Relations

Follow references to load related entities in a single query. No N+1 problem.

typescript 
// Load todos with their owners
{ todos: { owner: {} } }

// Load users with their todos and each todo's tags
{ users: { todos: { tags: {} } } }

// Filter nested relations
{ users: { todos: { $where: { done: false }, $limit: 5 } } }

Mutations

typescript 
// Create
db.transact(db.tx.todos[db.id()].set({
  title: 'Buy groceries',
  done: false,
  createdAt: Date.now()
}));

// Update (partial merge)
db.transact(db.tx.todos[existingId].merge({ done: true }));

// Delete
db.transact(db.tx.todos[existingId].delete());

// Link relations
db.transact(db.tx.users[userId].link({ todos: todoId }));

// Multiple operations in one transaction
db.transact([
  db.tx.todos[id1].merge({ done: true }),
  db.tx.todos[id2].delete(),
  db.tx.users[userId].unlink({ todos: id2 })
]);

Real-Time Sync

Every query in DarshanDB is a live subscription by default. When data changes anywhere, every connected client with a matching query receives the diff in under 1ms. No polling, no manual invalidation, no extra code.

How It Works

Real-Time Data Flow
Client A subscribes
Client B mutates
↓ ↓
WebSocket + MsgPack
↓
Query Engine
Mutation Engine
Sync Engine
↓
Permission Engine (RLS)
↓
Triple Store (PostgreSQL)
↑
Delta Diff → Push to Client A

Key Features

  • Persistent WebSocket with multiplexed subscriptions over a single connection
  • Reactive queries -- every useQuery is automatically a live subscription
  • Optimistic mutations -- instant UI updates, server reconciliation, auto-rollback on conflict
  • Offline-first -- IndexedDB persistence, operation queue, automatic sync on reconnect
  • Delta compression -- only changed fields are transmitted, not the full object
  • Catch-up protocol -- reconnecting clients receive only the diff since their last transaction

WebSocket Protocol

javascript 
const ws = new WebSocket('ws://localhost:7700/ws');

// 1. Authenticate
ws.send(JSON.stringify({ type: 'auth', token: 'ACCESS_TOKEN' }));

// 2. Subscribe to a query
ws.send(JSON.stringify({
  type: 'subscribe',
  id: 'sub-1',
  query: { todos: { $where: { done: false } } }
}));

// 3. Receive initial data
// { type: "q-init", id: "sub-1", data: { todos: [...] }, tx: 42 }

// 4. Receive live updates (delta only)
// { type: "q-diff", id: "sub-1", added: [], updated: [...], removed: [], tx: 43 }

// 5. Unsubscribe
ws.send(JSON.stringify({ type: 'unsubscribe', id: 'sub-1' }));

Server Functions

Server functions are TypeScript files in darshan/functions/ that run on the server inside V8 sandboxes with full database access. Four types cover every backend need.

Query -- Read-only, Cacheable, Reactive

typescript 
import { query, v } from '@darshan/server';

export const getTodos = query({
  args: { listId: v.id(), status: v.optional(v.string()) },
  handler: async (ctx, { listId, status }) => {
    const filter = { listId };
    if (status) filter.status = status;
    return ctx.db.query({ todos: { $where: filter } });
  },
});

Mutation -- Transactional ACID Writes

typescript 
import { mutation, v } from '@darshan/server';

export const createTodo = mutation({
  args: {
    title: v.string().min(1).max(500),
    listId: v.id(),
  },
  handler: async (ctx, { title, listId }) => {
    const todoId = ctx.db.id();
    await ctx.db.transact([
      ctx.db.tx.todos[todoId].set({
        title, done: false, listId,
        userId: ctx.auth.userId,
        createdAt: Date.now(),
      }),
    ]);
    return todoId;
  },
});

Action -- Side Effects

typescript 
import { action, v } from '@darshan/server';

export const sendWelcomeEmail = action({
  args: { userId: v.id() },
  handler: async (ctx, { userId }) => {
    const user = await ctx.db.query({ users: { $where: { id: userId } } });
    await ctx.fetch('https://api.sendgrid.com/v3/mail/send', {
      method: 'POST',
      headers: { Authorization: `Bearer ${process.env.SENDGRID_KEY}` },
      body: JSON.stringify({ to: user.email, subject: 'Welcome!' }),
    });
  },
});

Scheduled -- Cron Jobs

typescript 
import { scheduled } from '@darshan/server';

export const dailyCleanup = scheduled({
  cron: '0 3 * * *',  // 3 AM daily
  handler: async (ctx) => {
    const cutoff = Date.now() - 30 * 86400000;
    const stale = await ctx.db.query({
      sessions: { $where: { lastUsedAt: { $lt: cutoff } } }
    });
    for (const s of stale.sessions) {
      await ctx.db.transact(ctx.db.tx.sessions[s.id].delete());
    }
  },
});

Presence

Built-in real-time presence for tracking online users, cursors, typing indicators, and arbitrary ephemeral state. Presence data lives in memory -- it is not persisted to the database.

tsx 
import { usePresence } from '@darshan/react';

function CollaborativeEditor() {
  const { peers, updatePresence } = usePresence('document-123', {
    name: currentUser.name,
    cursor: null,
    isTyping: false,
  });

  const handleMouseMove = (e) => {
    updatePresence({ cursor: { x: e.clientX, y: e.clientY } });
  };

  return (
    <div onMouseMove={handleMouseMove}>
      {peers.map((peer) => (
        <RemoteCursor
          key={peer.id}
          name={peer.data.name}
          position={peer.data.cursor}
        />
      ))}
    </div>
  );
}

Authentication

DarshanDB includes a complete auth system. No third-party services required. Passwords are hashed with Argon2id (memory=64MB, iterations=3, parallelism=4). Accounts lock after 5 failed attempts for 30 minutes.

Permissions

DarshanDB uses a zero-trust model: everything is denied unless explicitly allowed. Define permissions in darshan/permissions.ts using a TypeScript DSL.

Permission Evaluation Flow
Client Query
↓
Auth Check (JWT verified?)
↓
Role Resolution (admin / editor / viewer)
↓
Row-Level Security (WHERE clause injected)
↓
Field Filtering (restricted fields stripped)
↓
Filtered Result Returned

Permission DSL

typescript 
// darshan/permissions.ts
export default {
  todos: {
    // Only the owner can read their todos
    read: (ctx) => ({ userId: ctx.auth.userId }),

    // Any authenticated user can create
    create: (ctx) => !!ctx.auth,

    // Only the owner can update
    update: (ctx) => ({ userId: ctx.auth.userId }),

    // Only admins can delete
    delete: (ctx) => ctx.auth.role === 'admin',
  },

  users: {
    read: {
      allow: true,
      fields: {
        email: (ctx, entity) => entity.id === ctx.auth.userId,
        passwordHash: false,  // never exposed
      },
    },
  },
};

Permission Rule Types

Rule TypeExampleBehavior
Booleantrue / falseAllow or deny all
Auth check(ctx) => !!ctx.authRequire authentication
Filter object(ctx) => ({ userId: ctx.auth.userId })Row-level filtering (becomes SQL WHERE)
Role check(ctx) => ctx.auth.role === 'admin'Role-based access control
Field restrictionfields: { email: false }Hide specific fields from response
⚠

Zero-trust default: When no permissions are defined for an entity, DarshanDB denies all access. You must explicitly allow every operation. Unauthorized data never leaves the database -- it is filtered at the SQL level, not after fetching.

Security Layers

11 layers of defense-in-depth, evaluated in order on every request:

  1. TLS 1.3 -- No plaintext, no TLS 1.2 fallback
  2. Rate Limiting -- Token bucket per IP/user/API key
  3. Input Validation -- Schema-validated at API boundary
  4. Authentication -- JWT RS256 + refresh + device binding
  5. Authorization -- Permission engine on every request
  6. Row-Level Security -- SQL WHERE injection, data invisible not forbidden
  7. Field Filtering -- Restricted fields stripped from response
  8. Query Complexity -- Rejects expensive queries
  9. V8 Sandboxing -- Functions isolated from system
  10. Audit Logging -- Every mutation: actor + timestamp + diff
  11. Anomaly Detection -- Unusual patterns trigger alerts

DarshanDB is built for speed at every layer. A persistent WebSocket with binary MsgPack encoding and delta-only patches eliminates the overhead of traditional REST APIs.

Tuning Guide

bash 
# Connection pool (default: 10)
DARSHAN_PG_POOL_SIZE=20

# Query complexity limits
DARSHAN_MAX_QUERY_DEPTH=8        # max nesting depth (default: 12)
DARSHAN_MAX_QUERY_RESULTS=5000   # max entities per result (default: 10000)

# Rate limits
DARSHAN_RATE_LIMIT_AUTH=200      # auth'd requests/min (default: 100)
DARSHAN_RATE_LIMIT_ANON=10       # anon requests/min (default: 20)

# WebSocket
DARSHAN_MAX_CONNECTIONS=50000    # max concurrent (default: 10000)

# Query cache
DARSHAN_QUERY_CACHE_SIZE=5000    # LRU entries (default: 1000)

Production Checklist

  • Set DARSHAN_PG_POOL_SIZE to 2x CPU cores
  • Enable PgBouncer for multi-server deployments
  • Set DARSHAN_MAX_QUERY_DEPTH to your app's minimum
  • Configure rate limits for auth and anon traffic
  • Monitor /metrics with Prometheus + Grafana
  • Set RUST_LOG=warn in production
  • Set up automated database backups
