# Migration Guide

This guide covers upgrading between DarshanDB versions and managing database schema changes.

## Upgrading DarshanDB

### Check Your Current Version

```bash
darshan --version
```

### Update the Binary

```bash
# Update to latest
curl -fsSL https://darshandb.dev/install | sh

# Or with Docker
docker pull ghcr.io/darshjme/darshandb:latest
docker compose up -d
```

### Update Client SDKs

```bash
# Update all DarshanDB packages
npm update @darshan/react @darshan/nextjs @darshan/client

# Or update individually
npm install @darshan/react@latest
```

## Version Compatibility

The server and client SDKs follow semantic versioning. The compatibility rules are:

| Server Version | Compatible Client Versions |
|---------------|---------------------------|
| `0.1.x` | `0.1.x` |
| `0.2.x` | `0.1.x`, `0.2.x` |
| `1.x.x` | `1.x.x` (stable API) |

**Rule of thumb:** Minor version bumps on the server are backward-compatible with older clients. Major version bumps may require client updates.

## Database Migrations

### Development Mode

In development (`darshan dev`), DarshanDB uses schema-on-read. The database schema adapts automatically as you write data -- no migrations needed.

### Production Mode

In production, DarshanDB requires explicit migrations to change the schema. This prevents accidental data loss.

#### Generate a Migration

```bash
# Compare your current schema definition against the database
darshan migrate generate --name add-priority-to-todos
```

This creates a migration file in `darshan/migrations/`:

```
darshan/migrations/
  20260405_000001_add_priority_to_todos.sql
```

#### Review the Migration

```sql
-- darshan/migrations/20260405_000001_add_priority_to_todos.sql
-- Auto-generated by DarshanDB

-- Up
ALTER TABLE triples ADD COLUMN IF NOT EXISTS priority_index INTEGER;
CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_triples_priority ON triples (priority_index);

-- Down
DROP INDEX IF EXISTS idx_triples_priority;
ALTER TABLE triples DROP COLUMN IF EXISTS priority_index;
```

Always review auto-generated migrations before applying them.

#### Apply Migrations

```bash
# Apply all pending migrations
darshan migrate up

# Apply a specific number of migrations
darshan migrate up --steps 1

# Dry run (show SQL without executing)
darshan migrate up --dry-run
```

#### Roll Back

```bash
# Roll back the last migration
darshan migrate down --steps 1

# Roll back all migrations
darshan migrate down --all
```

#### Check Status

```bash
# Show migration status
darshan migrate status

# Output:
# Migration                                    Status     Applied At
# 20260401_000001_initial                      Applied    2026-04-01 12:00:00
# 20260405_000001_add_priority_to_todos        Pending    -
```

### Strict Mode

Enable strict mode to enforce schema validation in production:

```bash
DARSHAN_STRICT_MODE=true
```

With strict mode enabled:
- All entity attributes must be declared in the schema
- Writing an undeclared attribute returns an error
- Migrations are required for any schema change

#### Defining Your Schema

```typescript
// darshan/schema.ts
import { defineSchema, defineTable, v } from '@darshan/server';

export default defineSchema({
  todos: defineTable({
    title: v.string(),
    done: v.boolean(),
    priority: v.number().optional(),
    createdAt: v.number(),
    userId: v.id(),
  }).index('by_user', ['userId'])
    .index('by_priority', ['priority']),

  users: defineTable({
    name: v.string(),
    email: v.string(),
    role: v.union(v.literal('admin'), v.literal('editor'), v.literal('viewer')),
  }).index('by_email', ['email']),
});
```

## Breaking Changes by Version

### 0.1.x to 0.2.x

No breaking changes planned for early releases. The 0.x series is pre-stable and may include breaking changes in minor versions. Pin your versions:

```json
{
  "dependencies": {
    "@darshan/react": "~0.1.0"
  }
}
```

### Pre-1.0 to 1.0 (Future)

The 1.0 release will freeze the public API. A detailed migration guide will be published when 1.0 is released.

## Backup Before Upgrading

Always back up your database before upgrading:

```bash
# Create a backup
darshan backup --output /backups/darshan-pre-upgrade-$(date +%Y%m%d).sql.gz

# Verify the backup
darshan backup verify --input /backups/darshan-pre-upgrade-20260405.sql.gz
```

## Troubleshooting

### Migration fails with "relation already exists"

This happens when a migration was partially applied. Check the migration state and fix manually:

```bash
# Check which migrations have been applied
darshan migrate status

# Mark a migration as applied without running it
darshan migrate resolve --applied 20260405_000001_add_priority_to_todos
```

### Client SDK version mismatch

If you see `PROTOCOL_VERSION_MISMATCH` errors:

```bash
# Check server protocol version
curl http://localhost:7700/api/admin/health

# Update your client SDKs to match
npm install @darshan/react@latest
```

### Schema drift in production

If your database schema has drifted from the expected state:

```bash
# Generate a migration to reconcile
darshan migrate generate --name reconcile-schema --from-database

# Review carefully, then apply
darshan migrate up
```
